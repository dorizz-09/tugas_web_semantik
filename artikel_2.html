Definisi dan Cara Kerja HTTP dan HTTPS

Bagaimana protokol ini mengelola pengiriman data antara server dan browser, serta pentingnya keamanan dalam HTTPS.

Bagaimana cara kerja HTTP? HTTP adalah singkatan dari HyperText Transfer Protocol, yaitu protokol sisi aplikasi yang digunakan untuk mentransmisikan dokumen hypermedia seperti HTML. HTTP mentransfer data melalui koneksi TCP antara klien dan server (Kita tidak akan membahas terlalu dalam tentang ini, kita bisa menambahkan blog baru tentangnya). HTTP mentransfer teks biasa melalui koneksi yang bisa dibaca oleh seseorang yang memiliki akses ke koneksi Anda. Misalnya, jika Anda bertukar data melalui wifi bersama, seseorang dapat mengendus data Anda menggunakan alat pengendus data (akan ditambahkan blog tentang cara menggunakan Wireshark untuk mengendus data). Mari kita batasi informasi tentang HTTP sampai di sini saja.

Sekarang mari kita alihkan fokus ke cara kerja HTTPS. HTTPS berarti HTTP di atas lapisan aman sehingga tidak ada yang bisa mengendus atau memodifikasi data di tengah (serangan Man in the Middle). Dalam HTTPS, saat klien mencoba terhubung dengan server, server akan mengirimkan sertifikat SSL. Sertifikat ini berisi kunci publik dan tanda tangan digital. Ini digunakan untuk memverifikasi apakah sertifikat valid atau tidak. Kunci publik akan digunakan untuk proses enkripsi asimetris. Pertama, mari kita pahami enkripsi asimetris dan simetris.

Enkripsi Asimetris

Enkripsi asimetris dikenal sebagai kriptografi kunci publik. Dalam sistem ini, kunci publik digunakan untuk mengenkripsi data, sementara kunci privat digunakan untuk mendekripsi data.

Enkripsi Simetris

Enkripsi simetris adalah jenis enkripsi di mana hanya satu kunci yang digunakan untuk mengenkripsi dan mendekripsi data. Entitas akan berbagi kunci selama komunikasi untuk mengenkripsi dan mendekripsi data.

Bagaimana HTTPS menggunakannya?

  1. Selama proses handshake, server mengirimkan sertifikat SSL yang memiliki kunci publik asimetris ke klien. Server juga memiliki kunci privat yang disimpan di server web (server sendiri).
  2. Klien kemudian akan membuat kunci sesi berdasarkan algoritma tertentu. Kunci sesi ini akan dienkripsi menggunakan kunci publik, kemudian dikirim ke server.
  3. Server akan menggunakan kunci privat asimetris untuk mendekripsi kunci sesi yang dienkripsi dan mendapatkan kunci sesi tersebut.
  4. Sekarang browser akan menggunakan kunci sesi untuk mengenkripsi dan mendekripsi data selama sesi tersebut. Ini disebut enkripsi simetris. Data kini aman karena hanya klien dan server yang mengetahui kunci sesi tersebut. Setelah sesi berakhir, proses ini akan diulang lagi dari langkah 1, karena kunci sesi tidak akan berlaku lagi.

Membajak kunci sesi akan sulit karena kunci tersebut hanya berlaku untuk waktu yang sangat singkat. Enkripsi simetris relatif lebih cepat dibandingkan enkripsi asimetris karena membutuhkan perhitungan yang sangat sedikit.

Beberapa algoritma enkripsi kunci publik yang populer adalah RSA (Rivest–Shamir–Adleman) dan ECC (Elliptic Curve Cryptography).

Setelah proses di atas selesai, semuanya akan sama seperti HTTP, yaitu berbagi data melalui koneksi TCP.

Apakah SSL dan TLS adalah hal yang sama?

SSL adalah pendahulu langsung dari protokol lain yang disebut TLS (Transport Layer Security). Pada tahun 1999, Internet Engineering Task Force (IETF) mengusulkan pembaruan terhadap SSL. Karena pembaruan ini dikembangkan oleh IETF dan Netscape tidak lagi terlibat, nama tersebut diubah menjadi TLS. Perbedaan antara versi terakhir SSL (3.0) dan versi pertama TLS tidak terlalu drastis; perubahan nama dilakukan untuk menandai perubahan kepemilikan.

Karena keduanya sangat erat terkait, kedua istilah ini sering digunakan secara bergantian dan membingungkan. Beberapa orang masih menggunakan istilah SSL untuk merujuk ke TLS, sementara yang lain menggunakan istilah "enkripsi SSL/TLS" karena SSL masih dikenal luas.
